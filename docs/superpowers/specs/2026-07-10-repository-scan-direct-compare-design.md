# 저장소 스캔 직접 비교와 규칙 출처 표시 설계

## 배경

현재 저장소 스캔 서비스에는 저장소별 기준선 스캔을 지정하고, 이후 스캔을 기준선과 비교하는 흐름이 있다. 이 방식은 운영자가 “어떤 스캔을 기준으로 삼았는지”를 계속 기억해야 하며, 사용자가 실제로 원하는 “두 스캔 결과를 직접 비교한다”는 흐름보다 추상적이다.

이번 변경은 기준선 중심 비교를 제거하고, 사용자가 저장된 스캔 결과 두 개를 직접 선택해 비교하는 방식으로 전환한다. 또한 스캔 결과만 보여주는 데서 끝나지 않고, 각 취약점 스캔 규칙이 어디에서 왔고 어떤 기준으로 탐지하는지 화면에서 확인할 수 있게 한다.

## 목표

- 기준선 지정, 기준선 필터, 기준선 우선 비교를 사용자 화면과 실행 흐름에서 제거한다.
- 사용자가 저장된 스캔 결과 두 개를 선택해 직접 비교할 수 있게 한다.
- 비교 결과에서 새로 발견된 취약점, 해결된 취약점, 계속 남아 있는 취약점, 검사 항목 변화를 보여준다.
- 각 스캔 규칙의 출처, 탐지 방식, 탐지 근거, 영향도, 필요 조치를 화면에서 확인할 수 있게 한다.
- 현재 스캐너가 외부 상용 스캐너나 취약점 데이터베이스가 아니라, 이 저장소 내부의 정적 분석 규칙 기반이라는 점을 명확히 표시한다.

## 제외 범위

- 기존 운영 데이터베이스의 기준선 테이블을 즉시 삭제하지 않는다.
- 외부 취약점 데이터베이스, 보안 업체 엔진, 실시간 위협 정보 연동은 이번 범위에 포함하지 않는다.
- 두 개를 초과하는 스캔을 한 번에 비교하는 기능은 포함하지 않는다.
- 취약점의 실제 악용 가능성을 동적 실행으로 검증하는 기능은 포함하지 않는다.

## 사용자 흐름

1. 사용자는 저장소 스캔 이력 화면에서 저장소별 스캔 목록을 확인한다.
2. 사용자는 비교할 스캔 두 개를 선택한다.
3. 화면은 두 스캔의 실행 일자, 저장소, 브랜치, 커밋, 스캔 식별자를 나란히 보여준다.
4. 화면은 지문 기반 비교 결과를 다음 그룹으로 보여준다.
   - 새로 발견된 취약점
   - 해결된 취약점
   - 계속 남아 있는 취약점
   - 통과 상태가 바뀐 검사 항목
5. 사용자는 각 취약점 카드에서 우선순위, 발견 위치, 발견 근거, 영향도, 필요 조치를 확인한다.
6. 사용자는 규칙 상세 영역에서 어떤 규칙이 어떤 파일의 어떤 분석 로직으로 동작하는지 확인한다.

## 비교 기준

비교는 발견 줄 번호가 아니라 발견 항목 지문을 기준으로 한다. 지문은 규칙 식별자, 파일 경로, 제목, 증거 문구를 정규화해 만든다. 줄 번호만 바뀐 경우에는 같은 취약점으로 볼 가능성을 높이고, 같은 취약점이 여러 번 발견된 경우에는 개수를 고려해 새 발견과 해결을 계산한다.

새 스캔 실행 직후의 자동 비교는 같은 저장소의 직전 스캔을 사용한다. 화면의 중심 기능은 사용자가 직접 선택한 두 스캔 비교다. 기준선이 있으면 우선 비교하는 기존 동작은 제거한다.

## 화면 설계

### 스캔 이력

- 기준선 지정 버튼을 제거한다.
- 기준선 필터를 제거한다.
- 각 스캔 카드에 `비교 A 선택`, `비교 B 선택` 동작을 제공한다.
- 같은 저장소의 최신 두 스캔이 있으면 초기 비교 대상으로 추천한다.
- 선택된 두 스캔이 서로 다른 저장소라면 비교를 막고, 같은 저장소 스캔을 선택하라는 안내를 보여준다.

### 비교 결과

- 상단에는 두 스캔의 요약 정보를 표시한다.
- 가운데에는 심각도별 변화 수치를 표시한다.
- 하단에는 다음 목록을 카드 형태로 표시한다.
  - 새로 발견
  - 해결됨
  - 계속 남음
  - 검사 항목 변화
- 각 취약점 카드는 기존 상세 카드 형식을 유지하되, 어느 쪽 스캔에 존재하는 항목인지 명확히 표시한다.

### 스캔 규칙과 분석 기준

- 규칙 목록은 별도 영역으로 표시한다.
- 각 규칙은 다음 정보를 가진다.
  - 규칙 식별자
  - 규칙 이름
  - 심각도
  - 분류
  - 탐지 방식
  - 탐지 요약
  - 구현 출처
  - 영향도
  - 필요 조치
  - 현재 사용 여부
- 구현 출처는 예를 들어 `src/lib/scanner/analyzers.ts`처럼 실제 코드 위치를 표시한다.
- 탐지 방식은 정규식 기반, 파일 경로 기반, 휴리스틱 기반, 프레임워크 구조 기반처럼 사람이 이해할 수 있는 용어로 표시한다.

## 자료 구조

기존 스캔 결과와 스캔 이력 구조는 유지한다. 비교 응답에서 기준선 관련 필드는 새 화면에서 사용하지 않는다. 장기적인 필드 삭제는 별도 운영 데이터 정리 작업으로 분리하고, 이번 작업에서는 운영 데이터 호환성을 위해 남겨둔다.

규칙 메타데이터에는 다음 정보를 추가한다.

- `sourcePath`: 규칙 구현 파일 경로
- `detectionType`: 탐지 방식
- `detectionSummary`: 탐지 조건 요약
- `impact`: 영향도 설명
- `remediation`: 조치 설명
- `limitations`: 알려진 한계 또는 오탐 가능성

## API 설계

기존 설정 조회 응답에 규칙 상세 메타데이터를 포함한다. 이번 구현에서는 별도 규칙 조회 API를 만들지 않고, 화면이 이미 사용하는 설정 조회 흐름을 확장한다.

스캔 직접 비교는 클라이언트가 저장된 두 스캔 결과를 가지고 계산한다. 서버 저장소에 없는 스캔을 비교하는 별도 비교 API는 이번 범위에 포함하지 않는다. 이번 범위에서는 현재 스캔 이력 응답에 포함된 저장 결과를 기준으로 구현한다.

## 오류 처리

- 비교할 스캔이 하나만 선택되면 두 번째 스캔을 선택하라는 안내를 표시한다.
- 서로 다른 저장소의 스캔 두 개가 선택되면 비교하지 않고 같은 저장소 스캔만 비교할 수 있다는 안내를 표시한다.
- 선택한 스캔이 삭제되면 선택 상태를 해제한다.
- 규칙 메타데이터가 누락된 경우에는 규칙 식별자와 기본 설명만 표시하고 화면 전체를 실패시키지 않는다.

## 테스트

- 기준선 지정 버튼과 기준선 필터가 화면에서 사라졌는지 검증한다.
- 기준선이 있어도 새 스캔 비교가 직전 스캔 기준으로 동작하는지 검증한다.
- 사용자가 두 스캔을 선택하면 새로 발견, 해결, 계속 남음 그룹이 표시되는지 검증한다.
- 서로 다른 저장소 스캔을 선택하면 비교가 차단되는지 검증한다.
- 규칙 상세 영역에 탐지 방식, 구현 출처, 영향도, 필요 조치가 표시되는지 검증한다.
- 스캔 규칙 메타데이터가 모든 규칙에 채워져 있는지 검증한다.

## 문서 갱신

`README.md`에서 기준선 중심 설명을 제거하고, 저장된 스캔 두 개를 직접 비교하는 흐름으로 갱신한다. 운영 설명에는 현재 스캐너가 내부 정적 분석 규칙 기반이라는 점과 규칙 출처를 화면에서 확인할 수 있다는 점을 추가한다.

## 전환 계획

현재 작업 중인 운영성 개선 풀 리퀘스트에 이 변경을 반영한다. 기준선 기능이 포함된 상태로 병합하지 않고, 직접 비교 중심으로 수정한 뒤 검증한다. 기존 데이터에 남아 있는 기준선 정보는 읽거나 표시하지 않으며, 이후 운영 데이터 정리 작업에서 삭제 여부를 따로 결정한다.
