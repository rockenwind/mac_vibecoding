# 저장소 스캔 규칙 품질 개선 설계

## 배경

저장소 스캔 서비스는 이미 스캔 실행, 결과 상세 카드, 과거 스캔 재조회, 비교, 삭제 흐름을 제공한다. 다음 단계는 결과 화면의 양보다 탐지 품질을 높이는 것이다. 현재 규칙은 비밀값, 위험한 명령 실행, 프롬프트 노출, MCP 설정 위험을 단순 정규식으로 찾는다. 1차 개선에서는 규칙 수를 늘리되, 제품 사용자가 바로 조치할 수 있도록 신뢰도와 구체적인 조치 안내를 함께 제공한다.

## 목표

- 실제 코드 점검에 자주 필요한 보안 규칙을 추가한다.
- 각 발견 항목에 신뢰도를 부여해 우선 확인 대상을 구분한다.
- 결과 카드와 보고서에 신뢰도를 함께 표시한다.
- 기존 스캔 이력, 비교, 삭제 기능과 호환되도록 저장 형식을 확장한다.
- 큰 구조 변경 없이 현재 스캐너의 책임 범위 안에서 구현한다.

## 제외 범위

- 외부 보안 도구 연동은 이번 범위에 포함하지 않는다.
- 추상 구문 트리 기반 정밀 분석은 이번 범위에 포함하지 않는다.
- 인공지능 기반 자동 수정 제안 생성은 이번 범위에 포함하지 않는다.
- 채용 서비스인 8000번 서비스와는 연결하지 않는다.

## 탐지 규칙

1차 규칙은 정적 텍스트 분석 기반으로 추가한다.

- 비밀값 노출: 기존 규칙을 유지하고 신뢰도 높음으로 표시한다.
- 위험한 명령 실행: `eval`, `new Function`, 셸 실행, 파이썬 하위 프로세스 호출을 탐지한다.
- 데이터베이스 질의 위험: 문자열 결합이나 템플릿 문자열로 만든 질의 실행을 탐지한다.
- 인증 누락 가능성: 공개 API 경로에서 인증 확인 없이 요청 처리만 있는 경우를 의심 항목으로 탐지한다.
- 관리자 기능 보호 누락 가능성: 관리자 경로 또는 관리자 함수명 주변에 권한 검사 흔적이 없으면 의심 항목으로 탐지한다.
- 위험한 외부 요청: 사용자 입력이 직접 주소나 요청 대상에 들어가는 패턴을 탐지한다.
- 넥스트 서버 비밀 노출: 클라이언트 구성 요소에서 서버 전용 환경 변수를 읽는 패턴을 탐지한다.

## 신뢰도 기준

신뢰도는 `high`, `medium`, `low` 세 값으로 저장한다.

- `high`: 실제 비밀값 형태, 명확한 위험 함수 사용처럼 코드 한 줄만으로도 위험성이 높은 경우
- `medium`: 위험한 조합이 강하게 의심되지만 주변 맥락 확인이 필요한 경우
- `low`: 이름, 경로, 단순 패턴상 검토가 필요한 경우

화면에는 한글 라벨인 `높음`, `중간`, `낮음`을 표시한다.

## 화면과 보고서

- 발견 카드에 `신뢰도` 항목을 추가한다.
- 기존 항목인 취약점, 우선순위, 발견 위치, 발견 근거, 영향도, 필요 조치는 유지한다.
- 마크다운 보고서에도 신뢰도를 포함한다.
- 정렬은 기존 심각도를 우선 유지하되, 같은 심각도에서는 신뢰도가 높은 항목을 앞에 둔다.

## 데이터 호환성

기존 저장된 스캔에는 신뢰도 값이 없을 수 있다. 화면과 보고서는 값이 없으면 `중간`으로 다룬다. 새 스캔 결과에는 항상 신뢰도를 저장한다.

## 테스트

- 스캐너 단위 테스트로 새 규칙과 신뢰도 값을 검증한다.
- 기존 비밀값 가림 처리가 유지되는지 확인한다.
- 결과 화면 테스트에서 신뢰도 라벨이 표시되는지 확인한다.
- 마크다운 보고서 테스트에서 신뢰도 항목이 포함되는지 확인한다.
- 타입 검사와 빌드를 실행해 저장 형식 확장이 기존 경로를 깨지 않는지 확인한다.
